# 对话整理

## 目标

将长对话整理成带证据、带确认状态、可审阅、可后续复用的结构化材料。

只做对话整理；不写 PRD，不写正式 Context，不替代专门文档 Skill。

## 方法

使用“简化 Gioia + 轻量 IBIS”。

- 简化 Gioia：按 `用户原话 / 对话片段 -> 一阶信号 -> 二阶主题 -> 整理结论` 整理，保证结论可追溯。
- 轻量 IBIS：遇到疑问、分歧、决策时，按 `问题 -> 候选观点 -> 理由/反对 -> 当前状态` 记录。

## 工作流程

1. 明确整理范围：说明纳入哪些对话、文件或材料，哪些不纳入。
2. 提取用户信号：按信号分类整理对话片段。
3. 合并主题：把相近信号归并为主题，保留关键原话和用户纠正。
4. 标注证据：区分用户原话、用户确认、AI 整理、AI 推断、待确认。
5. 输出整理稿：列出主题、结论、证据、待确认问题。
6. 等待确认：用户确认前不写文件；用户要求写入时，先给出拟写入路径和范围。

## 用户信号分类

- 方向判断：用户对业务、产品、技术、协作方式的判断。
- 问题与痛点：用户想解决的问题、当前卡点、反复出现的不适感。
- 想法与假设：用户提出的点子、可能方案、未验证设想。
- 决策与理由：用户已经拍板的选择，以及为什么这样选。
- 疑问与不确定性：用户还没想清楚、需要 AI 辅助判断或后续确认的问题。
- 偏好与边界：用户对结构、粒度、复杂度、写入位置、协作方式的偏好或限制。
- 情绪与强度：只记录会影响优先级、风险、协作方式的情绪信号；不做心理分析。
- 未分类：无法判断类别但可能有价值的信息，保留原始片段并放入待确认。

## 证据状态

- `用户原话`：用户明确说过的内容。
- `用户确认`：用户明确认可的判断、方向、命名、边界或决策。
- `AI整理`：AI 对多处用户表达的归纳，没有增加新含义。
- `AI推断`：AI 基于上下文推导出的内容，必须标为待确认。
- `待确认`：材料不足、含义不清或用户尚未确认。

## 输出规则

- 先输出整理稿，用户确认后才写文件。
- 整理稿不固定为正式文档模板，只保留内容维度。
- 每个关键结论尽量能追溯到原话、确认或对话片段。
- 冲突、分歧、用户纠正过的内容必须保留。
- 未确认内容、AI 推断、未分类片段必须显式标记。

## 写入边界

- 默认不写文件。
- 用户要求写入时，先说明拟写入路径、写入范围、是否新建或更新。
- 用户确认后，只写入 `drafts/`。
- 不直接写入 `workspace/` 或 `context/`。
- 如需迁移到 `workspace/` 或沉淀到 `context/`，交给后续确认流程和对应 Skill。
- 写成仓库文档时，文件名、一级标题、头部和更新记录交给文档规则 Skill，例如 `context-doc-rules`。

## 质量检查

- 是否说明整理范围。
- 是否覆盖用户信号分类，无法分类的是否进入“未分类”。
- 是否区分用户原话、用户确认、AI 整理、AI 推断、待确认。
- 是否保留用户纠正、分歧和被否定方向。
- 是否存在无来源强结论。
- 是否在写文件前等待用户确认。
