# Bug 管理

管理和维护当前项目的 bug 清单，覆盖从发现到关闭的完整生命周期。

## 核心文件

- bug 清单统一存放于当前项目的 `docs/bugs.md`
- 该文件与代码一起接受 git 版本化

## Bug 生命周期

```
┌──────────────┐                    ┌─────────┐    决定修复    ┌─────────┐    修复完成    ┌─────────┐
│ Claude 发现   │ ─┐                │  待修复  │ ──────────→ │  修复中  │ ──────────→ │  已关闭  │
└──────────────┘  │                └─────────┘              └─────────┘              └─────────┘
                  │   ┌─────────┐     │                          │
                  ├─→ │  待确认  │ ←───┤                          │
┌──────────────┐  │   └─────────┘     │                          │
│  用户自报     │ ─┘       │          │                          │
└──────────────┘           │          │                          │
                           ▼          │                          │
                    确认是设计/限制   │                          │
                           │          │                          │
                           ▼          ▼                          ▼
                    ┌──────────┐  ┌─────────┐              ┌─────────┐
                    │ 告知用户  │  │  不复现  │              │  不修了  │
                    │ 不记录    │  └─────────┘              └─────────┘
                    └──────────┘
```

## `docs/bugs.md` 格式

文件底部始终保留一段按示例格式的注释，确保不会因文件为空而丢失模板。

```markdown
# Bug 清单

> 最后更新：2026-06-23

## 概述

- 总计：5
- 待确认：0
- 待修复：2
- 修复中：1
- 已关闭：2
- 不复现：0
- 不修了：0

## Bug 列表

| ID | 发现日期 | 标题 | 严重程度 | 状态 | 复现步骤 | 模块/页面 | 关联 Commit | 关闭日期 |
|----|----------|------|----------|------|----------|-----------|-------------|----------|
| B001 | 2026-06-20 | 轮询导致候选池复选框被静默重置 | 严重 | 待修复 | 1. 后台任务运行中 2. 打开候选池 3. 取消所有活跃状态复选框 | 候选池 | — | — |
| B002 | 2026-06-21 | render() 无条件渲染隐藏页面 | 中等 | 修复中 | 1. 打开候选池 2. 后台任务运行时观察渲染调用 | app.js | — | — |
| B003 | 2026-06-18 | 分页切换时页码越界 | 轻微 | 已关闭 | 筛选后翻到超出范围的页码 | 任务列表 | a1b2c3d | 2026-06-20 |
| B004 | 2026-06-19 | 导入 BOSS JSONL 后统计未刷新 | 中等 | 不复现 | — | 导入页 | — | — |

## 已关闭 / 不复现 / 不修了

| ID | 发现日期 | 标题 | 严重程度 | 状态 | 复现步骤 | 模块/页面 | 关联 Commit | 关闭日期 |
|----|----------|------|----------|------|----------|-----------|-------------|----------|
| B003 | 2026-06-18 | 分页切换时页码越界 | 轻微 | 已关闭 | 筛选后翻到超出范围的页码 | 任务列表 | a1b2c3d | 2026-06-20 |
| B004 | 2026-06-19 | 导入 BOSS JSONL 后统计未刷新 | 中等 | 不复现 | — | 导入页 | — | — |

---

> 格式说明（以下为注释，创建 bug 时参考）：
> - **ID**：B + 三位数字序号，如 B001、B002。新 bug 取当前最大 ID + 1
> - **发现日期**：YYYY-MM-DD
> - **标题**：一句话简洁描述 bug 现象
> - **严重程度**：严重 / 中等 / 轻微
> - **状态**：待确认 / 待修复 / 修复中 / 已关闭 / 不复现 / 不修了
> - **复现步骤**：按照此步骤他人可复现该 bug（""表示无明确步骤）
> - **模块/页面**：bug 所在功能模块或页面名称
> - **关联 Commit**：修复该 bug 的 commit hash（修复后填写）
> - **关闭日期**：bug 关闭的日期，格式 YYYY-MM-DD（关闭时填写）
```

## 执行流程

### 阶段一：发现 Bug

当用户要求查找 bug 时，进入此阶段。

触发此阶段的词句包括："找一下 bug"、"帮我找 bug"、"检查一下有什么 bug"、"看看有没有 bug"、
"找找这个页面的 bug"、"这里面有什么问题"。

**执行步骤：**

1. 阅读与用户关注范围相关的代码文件（HTML、JS、服务端代码）
2. 关注以下方面的异常：
   - 状态被意外重置（如轮询刷新时的副作用）
   - 无用的 DOM 操作（如隐藏页面的渲染）
   - 无意义的函数参数（如死参数）
   - 内存泄漏风险（如定时器未清理、事件监听未移除）
   - 排序逻辑错误（如缺少稳定排序的 tie-breaker）
   - 并发/竞态：一个异步操作可能被另一个覆盖
   - 边界情况：空列表、单条记录、极端分页、输入为空等
3. 每条发现总结为：文件 + 行号 + 现象 + 触发条件
4. 以编号列表呈现给用户，格式：

```
### 发现 1：[简要概述]
- **文件**：path/to/file:line
- **现象**：具体发生了什么
- **触发条件**：何时会出现
- **严重程度**：严重 / 中等 / 轻微
```

5. 此阶段**不写入 `docs/bugs.md`**，只呈现给用户

### 阶段二：确认 Bug

发现阶段结束后，等待用户逐个确认哪些确属 bug。

用户确认方式通常为：
- "Bug 1 和 Bug 3 是真实的，记录下来" → 记录指定的 bug
- "第二个不是 bug" → 忽略
- "Bug 2 复现不了" → 可记录为"不复现"
- "全部确认" → 全部记录

只有经过用户确认的 bug 才写入 `docs/bugs.md`。

**写入步骤：**

1. 读取 `docs/bugs.md`（如文件不存在则创建）
2. 检查当前最大 ID 编号（如 B001、B002 等），为新 bug 分配连续编号
3. 用表格格式追加到"Bug 列表"区域
4. 更新"概述"部分的计数
5. 更新"最后更新"时间戳

### 用户自报 Bug

用户自己发现的 bug。与 Claude 发现的不同，这里不需要先做代码诊断——但**需要先判断它是否真的是一个 bug**（排除设计意图、环境问题等），判断通过后与其他 bug 一样进入"待确认"。

触发此阶段的词句包括："记录一个 bug"、"记录这个 bug"、"记个 bug"、"我发现了 bug"。

**执行步骤：**

1. 从用户的描述中理解问题
2. 阅读相关代码，判断是否真的是 bug（而非设计意图、环境限制、用户误解等）
3. 如果不是 bug，直接告诉用户原因，**不写入 `docs/bugs.md`**
4. 如果确实是 bug，写入 `docs/bugs.md`，状态为"待确认"，并告知用户等待确认
5. 从用户描述中提取：标题、严重程度、模块/页面、复现步骤
6. 如果用户没提严重程度，根据描述推断：
   - "体验很差""不方便""没有提示" → 轻微
   - "数据错了""丢失""异常""有时候不行" → 中等
   - "崩溃""不能用""死循环""安全" → 严重
7. 分配 ID，写入表格

### 阶段三：修复 Bug

当用户要求修复某个已记录的 bug 时，进入此阶段。

触发此阶段的词句包括："修复 bug X"、"改一下 B003"、"帮我修这个 bug"。

**执行步骤：**

1. 从 `docs/bugs.md` 读取 bug 详情
2. 将该 bug 状态从"待修复"改为"修复中"
3. 分析根因并实施修复
4. 修复完成后，在 commit message 中引用 bug ID：

```
fix: 轮询导致候选池复选框静默重置 (B001)
```

5. 将 bug 状态更新为"已关闭"，填写：
   - 关联 Commit：实际 commit hash（前 7 位）
   - 关闭日期：当天日期
6. 将已关闭条目移至"已关闭 / 不复现 / 不修了"表格

**一次性修复多个 bug 的规则：**
- 同一个 commit 可引用多个 bug ID：`fix: 候选池页面多项修复 (B001, B002)`
- 每个 bug 条目的"关联 Commit"填写同一个 hash

### 阶段四：查询与回顾

当用户想了解当前 bug 状况时，进入此阶段。

触发此阶段的词句包括："bug 列表"、"还有哪些 bug 没修"、"查看所有 bug"、"未修复的 bug"。

**执行步骤：**

1. 读取 `docs/bugs.md`
2. 简要报告：总计 / 待确认 / 待修复 / 修复中 / 已关闭
3. 如果用户问"还没修的"，只显示状态为"待修复"和"修复中"的条目

## 注意事项

- 仅在用户明确确认后才将 bug 写入 `docs/bugs.md`
- 修复 bug 必须在 commit message 中引用 bug ID
- 关闭 bug 时从活动列表移至"已关闭 / 不复现 / 不修了"表格，而非简单修改状态字段
- 如用户明确表示某个已记录 bug 不修了，状态记为"不修了"
- 如用户确认某个 bug 无法复现（尝试了复现步骤但无效），状态记为"不复现"，移至该表格
